# Deploying without Docker — Podman, bare metal, other proxies

The reference deployment is Docker Compose + Caddy, documented
end-to-end in [`deploy-linode.md`](./deploy-linode.md). This page is
for operators who can't or won't run Docker: rootless **Podman**,
plain **bare metal + systemd**, and a **reverse-proxy contract** for
running nginx / Traefik / Apache instead of Caddy. It closes with why
**Kubernetes** is a deliberate non-goal.

> **Who this is for.** An operator comfortable on a command line who
> has read [`deploy-linode.md`](./deploy-linode.md) once — this page
> reuses its DNS, firewall, governance, backup-discipline, and
> "before going public" sections rather than repeating them. Only
> the runtime changes here.

> **Support boundary.** The compose path is what
> `scripts/setup.sh` automates and what most operators run. The
> paths on this page are supported in the sense that the stack is
> designed to make them work and this doc is kept accurate — but
> you are the integration test for your host's quirks. Read §1
> first either way: everything else on this page falls out of it.

---

## 1. The deployment contract (what the stack actually is)

Understoria has no hard dependency on containers. The whole system is:

- **The community node** (`apps/server/`): one Node.js ≥ 20 process —
  `node dist/index.js` — a Fastify app listening on `HOST:PORT`
  (defaults `127.0.0.1:8787`), persisting to **one SQLite file** at
  `DATABASE_PATH`. All configuration is environment variables; the
  full table lives in [`operator-guide.md`](./operator-guide.md) and
  every variable is commented in `docker-compose.yml`. The one native
  dependency is `better-sqlite3-multiple-ciphers`, which compiles at
  `npm ci` time (needs `python3` + a C++ toolchain).
- **The PWA** (`apps/web/`): static files. `npm run build` produces
  `apps/web/dist/`; anything that can serve files over HTTPS can host
  it.
- **The proxy contract** (single-domain layout): serve the PWA at
  `/` with an SPA fallback to `index.html`, reverse-proxy `/api/*`
  to the node **with the `/api` prefix stripped**, and forward
  `X-Forwarded-For` so per-client rate limiting works (§4).
- **One secret**: `NODE_SYSTEM_SECRET_KEY`, generated by
  `scripts/generate-system-key.mjs` — a plain Node script with no
  Docker dependency.
- **Graceful shutdown**: the server handles `SIGTERM` itself. The
  container wraps it in `tini` only because Docker needs a PID-1
  signal forwarder; systemd and Podman deliver signals correctly
  without help.

If you remember one thing: *a supervised Node process, a directory
of static files, and a reverse proxy*. Everything below is one way
of arranging those three.

## 2. Podman (rootless) — the drop-in path

`docker-compose.yml` is standard Compose and runs under Podman. On
Debian/Ubuntu install `podman` + `podman-compose`; on Fedora/RHEL
`podman` ships with the `podman compose` provider. Rootless Podman is
the recommended shape — it's the same posture the compose file
already reaches for (`cap_drop: ALL`, `no-new-privileges`, read-only
rootfs), enforced one layer deeper.

Follow [`deploy-linode.md`](./deploy-linode.md) steps 1–2 and 4
unchanged, skip step 3 (Docker install), then use the **manual**
steps 6–8 with `podman compose` in place of `docker compose`.
`scripts/setup.sh` assumes the `docker` CLI and is not Podman-aware;
installing the `podman-docker` compatibility package (which shims
`docker` → `podman`) makes both the setup script and
`scripts/backup-db.sh` work unmodified, and is the simplest route.

Rootless differences that WILL bite if unhandled:

- **Ports 80/443.** Unprivileged processes can't bind below 1024.
  One sysctl fixes it host-wide:

  ```bash
  echo 'net.ipv4.ip_unprivileged_port_start=80' \
    | sudo tee /etc/sysctl.d/80-understoria.conf
  sudo sysctl --system
  ```

- **SELinux volume labels** (Fedora/RHEL/Alma). The Caddyfile bind
  mount needs a `:Z` suffix or Caddy reads a permission-denied:
  in `docker-compose.yml`, change
  `./deploy/Caddyfile:/etc/caddy/Caddyfile:ro` to
  `./deploy/Caddyfile:/etc/caddy/Caddyfile:ro,Z` (a compose-file
  edit you carry locally; Docker ignores the extra flag).
- **Restart-on-boot.** Rootless containers die with your login
  session unless you enable lingering and the restart service:

  ```bash
  sudo loginctl enable-linger "$USER"
  systemctl --user enable --now podman-restart.service
  ```

- **Logging options.** The compose file's `json-file` max-size
  options are Docker-specific; Podman logs to journald by default,
  which is better (journald already rotates). The unknown options
  are ignored with a warning — that warning is expected and safe.
- **Healthchecks + `depends_on` conditions** need Podman ≥ 4.x
  (Debian 12 and Fedora ship new enough; check
  `podman --version`).

Everything else — `.env`, key generation, smoke tests, redeploys,
the `down -v` data-loss warning — is identical to the reference
runbook. For backups without the `podman-docker` shim, set the
container runtime explicitly by editing the two `docker exec` /
`docker cp` calls in `scripts/backup-db.sh` to `podman`, or run the
bare-metal snapshot from §3.5 against the volume path
(`podman volume inspect understoria_understoria-data`).

## 3. Bare metal + systemd (no containers at all)

The shape for donated hardware, a spare laptop in a closet, or the
storm-hub machines in [`offline-resilience.md`](./offline-resilience.md)
— anywhere Docker is unavailable or unwanted. What follows was
verified end-to-end on a plain Linux host: build, key generation,
boot, `/health` + `/config`, and an online backup, all without a
container runtime.

### 3.1 Build prerequisites

Same as [`quickstart.md`](./quickstart.md) §1: `git`, Node **20+**
(22 recommended — it's what the containers pin), and a toolchain for
the SQLite native binding:

```bash
sudo apt-get update
sudo apt-get install -y git curl build-essential python3 nodejs npm
node --version   # want v20.x+; NodeSource one-liner in quickstart.md if older
```

### 3.2 Clone and build

```bash
sudo mkdir -p /opt && cd /opt
sudo git clone https://github.com/ArdentCascades/Understoria.git understoria
cd understoria
# Pin a tagged release for production, same as the reference runbook:
# sudo git checkout v0.3.0

# Install workspaces. The root postinstall builds packages/shared;
# better-sqlite3-multiple-ciphers compiles its native binding here.
sudo npm ci --workspaces --include-workspace-root --no-audit --no-fund

# Build the server and the PWA. On a 1 GB host, add swap first
# (deploy-linode.md §1) and raise V8's heap cap for the compile:
#   export NODE_OPTIONS=--max-old-space-size=1536
sudo npm --workspace @understoria/server run build
sudo npm --workspace @understoria/web run build
```

You now have `apps/server/dist/` (the node) and `apps/web/dist/`
(the PWA static bundle).

### 3.3 Generate the system key and write the env file

The key script is plain Node — no container needed:

```bash
node scripts/generate-system-key.mjs
```

Put the printed `NODE_SYSTEM_SECRET_KEY=` line into a root-owned env
file (this replaces the compose path's `.env`):

```bash
sudo install -m 600 -o root -g root /dev/null /etc/understoria/env
sudo nano /etc/understoria/env
```

Minimum contents — the same variables `docker-compose.yml` documents,
minus the container-specific ones:

```ini
NODE_ENV=production
HOST=127.0.0.1
PORT=8787
NODE_ID=node_yourcommunity
CORS_ORIGIN=https://understoria.example.org
# `loopback` (not `true`): the proxy connects from 127.0.0.1 on a
# bare-metal host, so this trusts X-Forwarded-For ONLY from local
# processes — tighter than the compose setting, per the comment in
# apps/server/src/config.ts.
TRUST_PROXY=loopback
NODE_SYSTEM_SECRET_KEY=<paste from the script>
OPERATOR_NAME="Example Mutual Aid Network"
OPERATOR_CONTACT=help@example.org
```

Do **not** set `DATABASE_PATH` here — the unit below pins it to the
systemd-managed state directory so it can't drift from the
`ReadWritePaths` sandbox.

`NODE_FOUNDER_KEYS`, `DATABASE_KEY`, peer/mirror settings, and the
disk-fill ceilings all work identically to the compose path; add
them to this file when the corresponding runbook section calls for
them.

### 3.4 The systemd unit

```bash
sudo useradd --system --home /var/lib/understoria --shell /usr/sbin/nologin understoria
sudo nano /etc/systemd/system/understoria.service
```

```ini
[Unit]
Description=Understoria community node
Documentation=https://github.com/ArdentCascades/Understoria
After=network-online.target
Wants=network-online.target

[Service]
User=understoria
Group=understoria
WorkingDirectory=/opt/understoria/apps/server
EnvironmentFile=/etc/understoria/env
# StateDirectory creates /var/lib/understoria owned by the service
# user and punches it through ProtectSystem=strict below — the ONLY
# writable path the process gets.
StateDirectory=understoria
Environment=DATABASE_PATH=/var/lib/understoria/understoria.db
ExecStart=/usr/bin/node /opt/understoria/apps/server/dist/index.js
Restart=on-failure
RestartSec=5

# Hardening — mirrors the container's posture (cap_drop ALL,
# no-new-privileges, read-only rootfs, private tmpfs /tmp).
NoNewPrivileges=yes
CapabilityBoundingSet=
ProtectSystem=strict
ProtectHome=yes
PrivateTmp=yes
PrivateDevices=yes
ProtectKernelTunables=yes
ProtectKernelModules=yes
ProtectKernelLogs=yes
ProtectControlGroups=yes
ProtectClock=yes
ProtectHostname=yes
ProtectProc=invisible
ProcSubset=pid
RestrictAddressFamilies=AF_INET AF_INET6 AF_UNIX
RestrictNamespaces=yes
RestrictRealtime=yes
RestrictSUIDSGID=yes
LockPersonality=yes
SystemCallArchitectures=native
SystemCallFilter=@system-service
UMask=0077
# Deliberately NOT set: MemoryDenyWriteExecute — V8's JIT needs
# writable-executable pages; enabling it kills Node at startup.

[Install]
WantedBy=multi-user.target
```

> **Node's path matters.** `ExecStart` assumes `/usr/bin/node` (what
> the Debian/NodeSource packages install — check with `which node`).
> An nvm-managed Node lives under a home directory, which
> `ProtectHome=yes` makes invisible to the service — the unit fails
> with "not executable: No such file or directory". Install Node from
> a system package for the service user rather than punching holes in
> the sandbox.

```bash
sudo systemctl daemon-reload
sudo systemctl enable --now understoria
sudo systemctl status understoria
curl -fsS http://127.0.0.1:8787/health   # {"status":"ok"}
```

`journalctl -u understoria` replaces `docker compose logs` —
journald's own rotation replaces the compose file's log caps.

### 3.5 Serve the PWA and proxy the API

Install Caddy natively (`sudo apt-get install caddy`, or the
[caddy repo](https://caddyserver.com/docs/install) for current
versions) and adapt `deploy/Caddyfile` — the shipped file is
container-flavoured in exactly three places:

1. `reverse_proxy understoria:8787` → `reverse_proxy 127.0.0.1:8787`
2. `root * /usr/share/caddy` → `root * /opt/understoria/apps/web/dist`
   (readable by the `caddy` user; `chmod -R o+rX apps/web/dist` after
   each build if your umask hides it)
3. The `log { output stdout }` block can be dropped — native Caddy
   under systemd already lands in journald.

Keep the `header { ... }` block **verbatim** — the CSP carries a
sha256 hash for the inline bootstrap script in `index.html`, and the
"refresh it with `scripts/csp-hash.sh` if you edit `index.html`"
rule applies to every proxy, not just the container one. Prefer
nginx or something else? Implement §4 instead.

### 3.6 Backups without Docker

Same principle as `scripts/backup-db.sh` (which is written against
`docker exec` and doesn't apply here): snapshot through SQLite's
`VACUUM INTO` using the server's own driver, never `cp` of a live
db file. Run as the service user so file ownership stays sane:

```bash
sudo -u understoria env DATABASE_KEY="$(sudo grep -oP '^DATABASE_KEY=\K.*' /etc/understoria/env || true)" \
  node --input-type=module - <<'JS'
import { createRequire } from "module";
const require = createRequire("/opt/understoria/apps/server/");
const Database = require("better-sqlite3-multiple-ciphers");
import fs from "fs";
const out = "/var/lib/understoria/.snapshot-tmp.db";
fs.rmSync(out, { force: true });
const db = new Database("/var/lib/understoria/understoria.db", { readonly: true });
const key = process.env.DATABASE_KEY;
if (key) db.pragma(`key = '${key.replace(/'/g, "''")}'`);
db.exec(`VACUUM INTO '${out}'`);
db.close();
JS
sudo mkdir -p /opt/understoria/backups
sudo mv /var/lib/understoria/.snapshot-tmp.db \
  "/opt/understoria/backups/understoria-$(date -u +%Y%m%dT%H%M%SZ).db"
sudo gzip -9 /opt/understoria/backups/understoria-*T*Z.db
```

Wrap that in a script + cron exactly as `deploy-linode.md` §10
describes, including the offsite step and the restore drill (restore
= stop the unit, gunzip the snapshot over
`/var/lib/understoria/understoria.db`, start the unit). The
`DATABASE_KEY` escrow warning applies unchanged.

### 3.7 Redeploys

```bash
cd /opt/understoria
sudo git fetch origin && sudo git checkout <new-tag>
sudo npm ci --workspaces --include-workspace-root --no-audit --no-fund
sudo npm --workspace @understoria/server run build
sudo npm --workspace @understoria/web run build
sudo systemctl restart understoria
```

Member data lives in `/var/lib/understoria`, which the build never
touches; the same "never delete the state" discipline as the compose
path's `down -v` warning applies to that directory.

## 4. Any reverse proxy: the contract

Caddy is bundled because TLS acquisition is one line, but nothing
depends on it. Whatever fronts the node must:

1. **Serve `apps/web/dist/` at `/`** with an SPA fallback — unknown
   paths return `index.html` (nginx: `try_files $uri /index.html;`),
   correct MIME types, and compression on.
2. **Proxy `/api/*` to the node with the prefix stripped** —
   `/api/posts` must arrive as `/posts`. nginx does this with a
   trailing slash on `proxy_pass`:

   ```nginx
   location /api/ {
       proxy_pass http://127.0.0.1:8787/;
       proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
       proxy_set_header Host $host;
   }
   ```

3. **Forward `X-Forwarded-For`** and set `TRUST_PROXY` accordingly
   (`loopback` for a same-host proxy, an IP/CIDR list for a separate
   proxy box). Skipping this silently collapses every member into
   one rate-limit bucket.
4. **Carry the security headers** from `deploy/Caddyfile`'s
   `header { ... }` block — HSTS, `nosniff`, referrer policy,
   permissions policy, and the CSP **including its sha256 script
   hash** (regenerate with `scripts/csp-hash.sh` after any
   `index.html` edit).
5. **Terminate TLS with a real certificate** (certbot, acme.sh,
   your call) — the PWA is service-worker based and will not
   install over plain HTTP.
6. Optionally health-check `GET /health` on the upstream, and cap
   request body size at or above the node's own 64 KB limit.

That's the entire interface. Traefik, Apache (`ProxyPass /api/
http://127.0.0.1:8787/`), HAProxy, or a CDN in front all work if
those six hold.

## 5. Kubernetes: a deliberate non-goal

Understoria's persistence is a **single-writer SQLite file**. That is
a feature — it is what makes a community node cheap enough to run on
a $5 VPS or a donated laptop — and it is fundamentally at odds with
what Kubernetes is for. Multiple replicas would corrupt the database
or fight over a ReadWriteOnce volume; a single replica on k8s is the
systemd unit from §3 with more moving parts and worse failure modes.

Resilience in this system comes from the **federation layer, not
process replication**: mirror nodes, client failover, seed vaults,
and community re-seed are designed, documented, and drilled in
[`community-resilience.md`](./community-resilience.md) and
[`community-reseed.md`](./community-reseed.md). Run one node per
community on the simplest runtime you can keep patched, and spend
the saved complexity budget on a mirror.

No manifests or Helm charts ship with the project, and issues asking
for multi-replica support will be redirected here.
