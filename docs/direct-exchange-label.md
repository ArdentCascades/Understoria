# Understoria — Direct-exchange label (design note)

> **Status:** **proposed** — design only; no implementation PRs yet.
> This is the follow-up design that shift-signups §14 ruling 1 filed:
> credit for help that has no post and no project to hang an exchange
> on. Read alongside `docs/shift-signups.md` §9 (the credit bridge and
> its wire trap — whose §9.2 permanent boundary this note must honor),
> `packages/shared/src/crypto.ts` (`verifyExchangeLabel` — the
> ingestion contract this note extends), and `apps/web/src/lib/
> timebank.ts` (the `project:`-prefix parsing this note must not
> confuse).

---

## §1 Status

Proposed. One genuinely open decision is marked "Operator ruling
needed" in §8, with a recommended default. Everything else here is
decided, pending ratification.

## §2 Why now

`Exchange.postId` is a required, signed, federated field. Today it
carries exactly two shapes:

- a real post id — the Board flow;
- `project:<projectId>/task:<taskId>` — project-task confirmations.

Help that arises OUTSIDE both shapes has no way to become credit:

- A potluck setup crew (a plain community event's shift) — the case
  shift-signups §14 ruling 1 deferred.
- Spontaneous help: "she saw me struggling with the fence and spent
  two hours on it." No post was ever written; making the pair
  retroactively author a fake NEED post just to record real hours is
  busywork that falsifies the record (the post would carry a made-up
  description, urgency, and expiry nobody meant).

The timebank's own values say this credit should be recordable:
`equal-time` (an hour is an hour, regardless of whether it was
scheduled through the Board) and honesty of the ledger (a fake
retro-post is worse data than an honest "direct" marker).

## §3 The label

```
direct:<uuid>
```

where `<uuid>` is freshly random at record time, generated by the
device that initiates the recording, with **no derivation from
anything** — not an event id, not a shift id, not a date, not a
member key. The uuid exists only to satisfy `postId`'s uniqueness
role in dedup/display; it is deliberately meaningless to every
observer.

**The §9.2 boundary, restated as this note's first constraint:**
`Exchange` federates. A label derived from an event or shift would
publish "member X exchanged hours in connection with gathering Y" —
the federated attendance graph `community-events.md` §11.1
permanently rejected, rebuilt through a side door. The `direct:`
namespace is the pressure-release valve that makes that boundary
livable: credit for gathering-adjacent help flows with NO gathering
correlator on the wire, because the label is random noise. The
threat-model §7 events entry already records the boundary; the
implementation PR extends that paragraph to name `direct:` as the
sanctioned alternative.

## §4 Wire + verification analysis

What actually changes on the wire: **nothing structural.** `postId`
is already an opaque signed string; `parseExchange` requires
non-empty and `verifyExchangeLabel` verifies signatures without
interpreting the label. A `direct:` exchange is byte-shape-identical
to today's rows. The work is in the CONSUMERS:

- **`verifyExchangeLabel` / ingestion scope** — unchanged
  mechanically, but the implementation PR adds a shared
  `isDirectExchangeLabel(postId)` helper and a test locking the
  namespace grammar (`direct:` + 36-char uuid, nothing else), so
  future consumers have one predicate instead of ad-hoc
  `startsWith` checks.
- **`/auto-confirm`** — a `direct:` exchange has no poster-signed
  post, so `bindToPost` already returns `post_not_found` and the
  system key will never finalize one. CORRECT and load-bearing:
  direct exchanges are mutual-signature-only, with no third-party
  finalization path. The implementation PR adds the explicit test.
- **`timebank.ts`** — `projectConfirmationOutflow` skips non-
  `project:` labels already; `balanceFor` never reads `postId`.
  Verified no-ops, locked by tests.
- **Dispute flow** — `disputeExchange` resolves the post behind the
  exchange to flip its status; a `direct:` exchange has none. The
  dispute proposal must render from the exchange's own fields
  (parties, hours, category, completedAt) with "recorded directly —
  no post" copy. Small, real work; named so it isn't discovered in
  production.
- **Display surfaces** — anywhere a post title is resolved from
  `postId` needs the same "recorded directly" fallback copy.

## §5 Consent is structural, not procedural

An `Exchange` requires BOTH `helperSignature` and `helpedSignature`
over the same canonical payload. That is the entire authorization
model, and `direct:` changes none of it: nobody can mint a direct
exchange naming a party who didn't sign. The recording flow reuses
the existing two-signature ceremony and the existing anti-gaming
safeguards (`assertWithinDailyLimit`, `evaluateSafeguards`) verbatim
— a direct exchange is MORE likely to trip the reciprocal-pattern
flag, which is correct and wanted.

What `direct:` removes is only the *artifact* requirement, not any
consent or verification step.

## §6 Entry points (phase 1: exactly two)

1. **The plain-event shift credit affordance** (`shift-signups.md`
   §9.3's gap): a passed shift on a plain event gains the same quiet
   "Record time together" link work-day shifts have, landing on the
   direct-exchange form with hours defaulting to the shift duration
   (editable; the helper states actuals per `equal-time`) and
   category from the event. Prefill in the FORM only — the resulting
   record carries the random `direct:` uuid and nothing event-shaped.
2. **A member's profile page**, visible to the OTHER party of an
   in-person moment: "Record time together" on the profile of the
   member you helped / who helped you. Both taps land on one form;
   both parties sign per §5.

No Board surface, no feed, no suggestion engine. Recording direct
credit is something two people who already shared the hours walk up
and do.

## §7 What this is NOT

- **Not a private-exchange channel.** A `direct:` exchange federates
  exactly like every exchange: parties, hours, category, timestamp
  are public wire data. The form says so before signing (the same
  comparison-card discipline as everything else).
- **Not an event/shift pointer.** Permanent boundary, §3. A future
  proposal wanting gathering-linked credit on the wire supersedes
  `community-events.md` §11.1, not this note.
- **Not a bypass of safeguards.** §5 — same flags, same daily limit,
  same dispute exposure.
- **Not retroactive bulk entry.** One exchange per ceremony; no
  import surface. `completedAt` keeps the existing not-far-future
  bound; honest backdating of a last-week moment is permitted
  exactly as the post flow permits it.

## §8 Open question — Operator ruling needed

1. **Should the profile entry point (§6.2) ship in phase 1, or only
   the shift affordance?** The profile surface makes direct credit
   general; the shift surface alone keeps it scoped to gatherings.
   **Recommended default: ship both.** The consent floor is the
   mutual signature (§5), which does not weaken with generality, and
   the spontaneous-help case (§2) is real mutual aid the ledger
   currently refuses to see. If pilot signal shows the profile
   surface inviting awkward pressure ("sign this"), narrowing is a
   one-line UI removal with no wire consequence.

## §9 Implementation phases

- **PR B — shared:** `isDirectExchangeLabel` + namespace-grammar
  tests; consumer no-op locks (`timebank.ts`, `balanceFor`);
  `/auto-confirm` explicit `post_not_found` test for `direct:`.
- **PR C — client:** the direct-exchange form + two-signature
  ceremony (reusing the confirm-flow signing mechanics unchanged);
  safeguards wiring; dispute + display fallback copy ("recorded
  directly — no post"); i18n en/es.
- **PR D — server: LOUDLY SKIPPED.** No wire change (§4); the label
  is already opaque to ingestion.
- **PR E — entry points:** the plain-event shift affordance + the
  profile affordance (per the §8 ruling); threat-model §7 paragraph
  extension naming `direct:` as the sanctioned no-correlator path.
